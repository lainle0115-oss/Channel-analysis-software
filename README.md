# Retail Channel AI Assistant

面向零售渠道销售与运营团队的多渠道分析工作台。它不要求所有文件采用同一模板，
而是自动寻找 Excel 数据表头、推断字段语义，并允许用户逐文件确认或修正映射。

## 当前能力

- 自动寻找 Excel 中最可能的数据工作表和表头行
- 根据列名语义推断日期、SKU、城市、门店、销量、销售额和库存
- 根据文件名和字段特征识别盒马、小象等渠道
- 每个上传文件都可单独修正渠道、数据类型和字段映射
- 保留来源文件、渠道、SKU、日期、城市和门店粒度
- 自动移除重叠周报/月报中的重复业务记录
- 盒马销售数据按“经营店名称含店”清洗，排除前置仓、FDC 等非门店记录
- 当前库存按“渠道 + SKU”的最新可用日期快照计算，不跨日期叠加每日库存
- 小象库存合计大仓、前置站点和在途库存；盒马库存使用最新“库存总数”
- 按渠道对比销量、销售额、SKU 数、区域数和最新库存
- 支持按渠道、来源文件和 SKU 切换交互趋势视角
- 单独查看每个渠道和每个 SKU 的趋势与排名
- 按“渠道 + SKU”识别缺货、库存不足和销量变化
- 生成日报、自然周周报和自然月月报，并与上一同期对比
- 自动获取上海、苏州当前天气与未来 7 天高温/强风风险，并写入经营报告
- 将天气、节假日和渠道活动上下文纳入原因假设与行动建议
- 支持销售 Pipeline 维护，可选择是否写入日报、周报和月报
- 导出简洁商务 Word 经营报告，包含核心指标、管理层摘要、渠道表现、TOP SKU、行动跟进和口径说明
- 报告中的行动跟进表包含负责人、截止日期和处理状态口径，可直接用于业务复盘
- 对销售额缺失和周期数据不完整进行明确提示，不把缺失数据包装成完整结论
- 展示文件识别、标准化来源和数据质量结果

## 快速运行

本地开发默认直接使用 Python 虚拟环境启动 Streamlit，不依赖 Docker / OrbStack。默认端口为
`8502`，用于避开本机可能被占用的 `8501`。

```bash
./scripts/start_app.sh
```

浏览器访问 `http://127.0.0.1:8502/`。上传文件会持久化到 `.streamlit/uploaded_files/`。
常用本地服务命令：

```bash
./scripts/status_app.sh
./scripts/restart_app.sh
./scripts/stop_app.sh
```

如果需要临时换端口：

```bash
PORT=8503 ./scripts/start_app.sh
```

本地确认效果后，再提交并推送到 GitHub，Render 会自动部署线上版本。

线上部署建议配置访问口令：

- 在 Render 的 Environment 中新增 `APP_PASSWORD`。
- 配置后，访问页面会先要求输入口令；本地未设置该环境变量时默认不拦截，便于开发调试。
- 可选配置 `MAX_UPLOAD_BYTES` 调整后端单文件上限；默认 200MB。Streamlit 配置层也限制为 200MB。

运行测试：

```bash
.venv/bin/python -m pytest -q
```

## 使用流程

1. 上传一个或多个 Excel / CSV 文件。
2. 在左侧展开每个文件，确认自动识别的渠道、数据类型和字段映射。
3. 使用日期、渠道和 SKU 筛选器控制分析口径。
4. 在“渠道对比”和“SKU 分析”中下钻。
5. 在“报告中心”选择日报、自然周周报或自然月月报，确认自动天气、补充经营上下文并下载 Word 报告。
6. 在“数据质量”中检查识别结果和标准化数据。

## 业务演示

使用内置示例数据即可完成一次 5-8 分钟演示。具体点击路径、讲解话术、示例上下文、
预期结果和边界说明见 [docs/DEMO_RUNBOOK.md](docs/DEMO_RUNBOOK.md)。

当前版本应定位为“可解释的零售渠道经营决策助手 MVP”：数字、异常和预测由确定性代码计算，
报告文字由规则模板生成。已支持 Render 上线口令门禁，但还不是多用户账号系统；尚未接入真实 LLM、
数据库、自动推送、定时任务和用户级数据隔离。

## 标准数据模型

| 字段 | 含义 |
|---|---|
| `date` | 数据日期 |
| `channel` | 渠道 |
| `source_file` | 来源文件 |
| `data_type` | 销售、销售库存、库存、采购或结算 |
| `sku_id` | SKU 编码 |
| `sku_name` | SKU 名称 |
| `city` | 城市或区域 |
| `store` | 门店或仓库 |
| `sales_qty` | 销量口径 |
| `sales_amount` | 销售额或被确认的金额口径 |
| `purchase_qty` | 采购订单数量，独立于销量 |
| `stock_qty` | 标准化库存快照 |

日期和销量、采购数量或库存至少需要识别其一才能进入分析。销售额、SKU、城市、门店或库存缺失时，
系统会降级处理并显示口径提醒，而不是直接阻断所有分析。

## 重要口径说明

- 自动推断不能替代业务确认。采购单、结算单中的“数量”和“金额”含义与销售不同，
  上传后应确认数据类型和字段映射。
- 采购订单会自动识别为“采购”数据类型，`采购数量`、`确认数量`、`送货数量`、
  `实收数量` 等字段进入 `purchase_qty`，不会计入销量或销售额。报告会输出销售数量、
  采购数量、采销比和 SKU 采销差。
- 当同时上传销售数据和采购订单时，采销分析只保留销售数据中出现过的 SKU；
  采购订单里的其他 SKU 会被排除，并在数据口径提醒里说明排除的 SKU 数和采购量。
- 上传文件会保存到本地 `.streamlit/uploaded_files/`，刷新网页后继续保留；
  只有点击侧边栏“清除已保存文件”才会删除。该目录已加入 `.gitignore`。
- 线上 Render 环境若没有配置 `APP_PASSWORD`，任何拿到链接的人都可能访问同一套上传文件；
  不要在未加口令的公开环境上传敏感销售数据、采购价或客户名单。
- 后端只接受 `.csv`、`.xls`、`.xlsx`，单文件默认不超过 200MB；非法后缀、空文件和超限文件会被拒绝。
- 销售额缺失时，系统继续分析销量和库存，但跨渠道销售额比较不完整。
- 销量和销售额可以在所选日期范围内累加；库存是日期快照。总览、渠道、SKU 和报告中的
  “当前库存 / 最新库存”均按每个“渠道 + SKU”的最新可用快照计算，绝不把每日库存跨日期相加。
- 小象单行库存由大仓库存、前置站点库存、供应商到大仓在途和大仓到门店在途合计；
  盒马单行库存使用“库存总数”。库存趋势图仍保留每日快照，用于观察库存变化。
- 异常规则是可解释的 MVP 规则，生产使用前需要结合补货周期和业务阈值调整。
- 当前完整周期预测按已覆盖日期日均外推，适合演示和初步判断，不是正式需求预测模型。
- 上海、苏州天气与未来 7 天高温/强风风险自动获取；强风/台风提示仅用于经营准备，正式预警以气象部门为准。
