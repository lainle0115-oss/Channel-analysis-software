# Retail Channel AI Assistant 演示落地稿

## 演示定位

一句话介绍：

> 把盒马、小象、叮咚等不同格式的渠道报表，统一成一套可追溯的数据口径，快速定位销售与库存异常，并输出业务团队可以继续跟进的经营报告。

这次演示要证明的不是“模型会聊天”，而是三个可落地能力：

1. 混乱渠道文件可以被稳定接入、检查和统一。
2. 系统能从渠道、SKU、库存三个层级找出需要行动的问题。
3. 分析结果可以沉淀为日报、周报、月报和 Word 经营简报，交给业务团队继续使用。

| 多渠道经营看板 | 可演示 | 总览、渠道对比、SKU 下钻、筛选和趋势 |

## 还缺的关键闭环

这些能力尚未完成，演示时不要表述为已上线：

| 缺口 | 为什么关键 | 下一步最小闭环 |
|---|---|---|
| 真实 LLM 分析 | 当前报告是可解释规则模板，不是大模型生成 | 让代码提供事实，LLM 只负责解释、追问和采销沟通文案 |
| 历史数据存储与 SQL | 当前分析依赖每次上传文件 | 增加 SQLite，保存标准化记录、文件批次和规则结果 |
| 业务反馈回写 | 当前行动建议没有负责人、状态和结果 | 增加异常确认、负责人、处理状态和结果记录 |
| 自动分发 | 当前需要手工下载 Word 报告 | 增加定时任务及飞书/企业微信推送 |
| 生产级预测 | 当前预测为已覆盖日期日均外推 | 引入促销、节假日、天气和历史季节性并回测 |
| 部署与治理 | 当前为本地 Streamlit | 增加 Docker、线上部署、权限、日志、密钥和数据脱敏 |

最关键的产品闭环是：

> 发现异常 → 指派负责人 → 执行动作 → 回写结果 → 验证销量和库存是否改善。

## 5-8 分钟演示路径

### 0:00-0:45 开场：先讲业务问题

页面保持默认“使用示例数据”。

讲解：

> 渠道团队每天收到的盒马、小象、叮咚报表格式都不同。原流程通常是人工复制、对字段、找异常，再写日报。这个助手先把接入和计算做成确定性流程，避免直接让 AI 猜数字。

### 0:45-1:40 证明数据可以接入和追溯

点击“数据质量”。

指出：

- 三个来源文件被识别为三个渠道。
- 原始字段名不同，但进入同一标准数据模型。
- 每条标准化数据仍保留来源文件、渠道、日期和 SKU。

讲解：

> 新渠道接入时，系统先自动推断，业务人员再确认映射。这样既降低接入成本，又不把自动识别当成绝对正确。

### 1:40-3:00 从总览找到经营问题

点击“经营总览”。

默认三天示例数据的预期结果：

- 3 个渠道，6 个 SKU。
- 总销量 740 件，销售额 11,397 元。
- 小象销量最高，三天累计 266 件。
- 系统识别 6 条待跟进异常。

重点指出：

- 小象“无糖酸奶200g”销量 61 件但库存为 0，属于缺货风险。
- 盒马“燕麦奶1L”销量明显增长但库存仅 18 件，需要确认补货承接。
- 盒马“低脂牛奶950ml”销量明显下降，需要检查价格、活动和上架状态。

讲解：

> 看板不是只告诉业务“卖了多少”，而是把销售变化和库存承接放在一起，直接形成下一步核查清单。

补充指出：

> 当前库存不是把每日库存相加。系统先找出每个“渠道 + SKU”的最新可用库存快照，再汇总到总览、
> 渠道和 SKU；每日库存只在趋势图中作为快照展示。

### 3:00-4:00 下钻一个渠道或 SKU

点击“渠道对比”中的盒马，或进入“SKU 分析”选择一个异常 SKU。

讲解：

> 管理层看到跨渠道问题后，运营可以继续下钻到渠道、SKU 和区域，而不需要重新整理原始表格。

### 4:00-6:30 生成业务可用报告

点击“报告中心”，选择“日报”和 `2026-06-10`。

指出该日总销量为 262 件、销售额为 4,124 元，并与 `2026-06-09` 对比。

填写并点击“应用报告上下文”：

```text
天气 / 温度 / 履约影响：
上海高温 32°C；北京与杭州闷热，乳饮和即食需求可能上升

节假日 / 特殊日期：
618 预热期，临近周末

最近渠道活动 / 价格 / 上架变化：
盒马燕麦奶资源位加码；叮咚冷萃咖啡促销；小象无糖酸奶正常销售
```

指出报告中的：

- 管理层摘要和上一日对比。
- 渠道表现、SKU 贡献、库存风险和行动清单。
- 外部上下文被标记为原因假设，而不是直接宣称因果。
- “下载 Word 日报”可以交给业务继续流转。

再切换“周报”或“月报”，只展示完整自然周期和缺失天数提示：

> 系统不会因为当前只上传了三天数据，就把三天包装成完整周报或月报；它会显示完整自然周期，并明确提示数据缺口。

### 6:30-7:30 收束：讲价值和下一步

讲解：

> 当前 MVP 已经把“文件接入、口径统一、异常识别、报告输出”串起来。下一步不是继续堆图表，而是补上负责人和处理结果回写，再接定时推送与 LLM 表达层，形成真正的经营闭环。

## 演示时的边界话术

业务方问“这是 AI 吗？”：

> 当前版本先把数字计算和异常判断做成可解释规则，避免模型编造事实。后续 LLM 会基于这些已计算事实生成更自然的解释、追问和沟通文案。

业务方问“为什么盒马没有销售额？”：

> 如果销售文件没有销售额或售价，系统不会用结算金额冒充 GMV，而是继续分析销量和库存，并明确标记金额口径不完整。

业务方问“预测准吗？”：

> 当前是按已覆盖日期日均外推的基线预测，用于发现节奏偏差；正式上线前需要加入历史季节性、活动、天气并做回测。

业务方问“异常阈值能改吗？”：

> 当前阈值是透明的 MVP 规则。上线时应按渠道、品类、补货周期配置，并记录业务确认结果持续校准。

业务方问“库存是不是把每日库存叠加了？”：

> 没有。销量和销售额可以跨日期累加，但库存是快照。当前库存按每个渠道和 SKU 的最新可用日期
> 取值后再汇总；小象会合计大仓、前置站点和在途库存，盒马使用最新库存总数。
